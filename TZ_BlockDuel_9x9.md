# Техническое задание и техническая документация
## Соревновательный блок-пазл «BlockDuel 9×9» (рабочее название)

**Версия документа:** 1.0
**Статус:** черновик для согласования
**Жанр:** competitive block puzzle (соревновательный «woodoku» с фигурами-тетромино)

---

## 1. Общие сведения

### 1.1. Краткое описание
Соревновательная головоломка на поле **9×9**. Игроки по очереди ставят на **общую доску** фигуры в форме тетромино (как в классическом тетрисе), генерируемые по правилу «мешка» (7-bag). Заполнение сплошной горизонтали, вертикали или квадрата 3×3 очищает эти клетки. Очки получает игрок, **чей ход вызвал очистку**. Партия идёт до тех пор, пока активный игрок не сможет поставить ни одну фигуру из своей руки («ходы кончились»). Побеждает игрок с большим количеством очков.

### 1.2. Цели продукта
- Простые, узнаваемые правила (Tetris + Sudoku-сетка) и высокая реиграбельность.
- Доступность: браузер, десктоп, мобильные устройства из одной кодовой базы.
- Соревновательный режим против бота (3 уровня), локально на одном экране и онлайн.

### 1.3. Целевая аудитория
Казуальные и полуказуальные игроки 9+; быстрые сессии 2–6 минут; кросс-платформенная аудитория (в т.ч. совместная игра вдвоём на одном устройстве — например, в дороге).

### 1.4. Глоссарий
| Термин | Значение |
|---|---|
| Доска (board) | Игровое поле 9×9 = 81 клетка |
| Бокс (box) | Один из 9 квадратов 3×3 |
| Фигура (piece) | Полимино в форме тетромино (I, O, T, S, Z, J, L) |
| Мешок (bag) | 7-bag рандомайзер: все 7 фигур по одному разу, затем перетасовка |
| Рука (hand) | Набор из K фигур, доступных игроку для хода |
| Ход (move) | Установка одной фигуры из руки на доску |
| Очистка (clear) | Удаление сплошной строки / столбца / бокса |
| Тупик (deadlock) | Ни одна фигура из руки не помещается на доску |
| PRNG | Псевдослучайный генератор с зерном (seed) для детерминизма |

## 2. Игровая механика (Core Gameplay)

### 2.1. Поле
- Сетка 9×9, координаты `(row, col)`, `row, col ∈ [0..8]`.
- Поле логически делится на 9 боксов 3×3. Индекс бокса: `box = (row // 3) * 3 + (col // 3)`.
- Состояние клетки: `empty` или `filled`. Опционально клетка хранит `ownerId` (кто поставил) — **только для подсветки/визуала**, на подсчёт очков не влияет.

### 2.2. Фигуры
- Базовый набор — 7 тетромино: `I, O, T, S, Z, J, L` (каждая занимает 4 клетки).
- Каждая фигура задаётся набором клеток-офсетов от опорной точки.
- **Поворот:** параметр `rotationEnabled` (по умолчанию `true`). При включении игрок может вращать фигуру (4 ориентации; O без эффекта). Реализация поворота — простая (вращение матрицы офсетов на 90°), без wall-kick.
- **Отражение (flip / зеркало):** параметр `flipEnabled` (по умолчанию `true`). Зеркальное отражение фигуры по горизонтали. Заметно влияет на S/Z/J/L; для I/O/T симметричные случаи дедуплицируются. Обе операции (поворот и отражение) — **настройки матча** и применяются к фигурам **в руке игрока** перед установкой.
- Расширяемость: набор фигур вынесен в конфиг — можно добавить «классические woodoku»-полимино (1×1, 1×2, 2×2, L-tromino и т.д.) в будущих режимах.

### 2.3. Генерация фигур — правило мешка (7-bag)
- Создаётся очередь: берутся все 7 фигур, перемешиваются детерминированным PRNG (зерно = `matchSeed`), выдаются по порядку; когда мешок пуст — формируется новый перемешанный мешок.
- **У каждого игрока — свой мешок** (`sharedBag = false`): независимый 7-bag со своим зерном (`seedA`, `seedB`). Каждый поток сам по себе честный (каждая фигура ровно 1 раз в мешке), руки игроков формируются независимо.
- Рука каждого игрока пополняется до `K` фигур (по умолчанию `K = 3`) из **собственного** мешка.

```
function nextPiece(bagState, rng):
    if bagState.queue is empty:
        bagState.queue = shuffle([I,O,T,S,Z,J,L], rng)   # детерминированно
    return bagState.queue.pop_front()
```

### 2.4. Ход
1. Активный игрок выбирает фигуру из своей руки и позицию на доске (опц. поворот).
2. **Валидация:** все клетки фигуры в пределах поля и пусты. Иначе ход невозможен.
3. Фигура фиксируется (клетки → `filled`).
4. **Очистка:** проверяются все строки, столбцы и боксы; полностью заполненные очищаются **одновременно** (см. 2.5).
5. **Начисление очков** игроку за этот ход (см. 2.6).
6. Рука игрока пополняется до `K` из общего потока.
7. Ход переходит сопернику.

### 2.5. Правила очистки
- Очищаются: любая полностью заполненная **строка** (9 клеток), **столбец** (9 клеток), **бокс** 3×3 (9 клеток).
- Все завершённые линии в рамках одного хода определяются **до** удаления и очищаются одновременно (важно для подсчёта пересечений и мульти-очисток).
- Клетка, входящая одновременно в очищаемые строку, столбец и бокс, очищается один раз; пересечения учитываются в бонусе за мульти-очистку.

### 2.6. Подсчёт очков (балансная таблица, конфигурируемо)
Пусть `N` — число очищенных «единиц» (строк + столбцов + боксов) за один ход.

| Очищено единиц `N` | Очки за ход (база) |
|---|---|
| 1 | 1 |
| 2 | 3 |
| 3 | 6 |
| 4 | 10 |
| n | n·(n+1)/2 |

Дополнительные модификаторы (включаемы флагами):
- **Комбо:** счётчик растёт за каждый ваш ход, который что-то очистил, и сбрасывается, если ваш ход не дал очистки. Множитель: `1 + 0.1 · min(combo, 10)`.
- **Perfect clear:** если после хода доска полностью пуста — `+15` очков.
- Итог за ход: `round(base · комбо_множитель) + бонусы`.

### 2.7. Условие окончания партии
- **Базовое (по умолчанию):** партия завершается, когда у активного игрока **ни одна фигура из руки не помещается** ни в одну позицию. → «ходы кончились».
- Альтернативные режимы (конфиг `endCondition`): «лимит ходов», «лимит времени», «до X очков». Реализуются тем же ядром.
- **Победитель:** больший счёт. **Ничья:** равный счёт → доп. критерии: меньшее число поставленных клеток / больше мульти-очисток / при онлайн — sudden death.

### 2.7.1. Таймер на ход (Blitz / цейтнот) — основной механизм конечности
Чтобы режим «до тупика» гарантированно завершался даже при сильной игре, на каждый ход даётся **ограниченное и сокращающееся время**:
- `turnTime` стартует с `turnTimeStart` (например, 12 с) и **уменьшается** каждый полный раунд на `turnTimeDecay` (например, −0.4 с), но не ниже `turnTimeMin` (например, 3 с).
- **На таймауте — принудительная постановка:** система ставит текущую выбранную фигуру (или случайную из руки) в **случайную валидную** позицию. Это ключевой момент: таймаут даёт постановку, а не пропуск, иначе доска перестаёт заполняться и партия не сходится к тупику.
- Эффект: цейтнот → ошибки и «мусорные» постановки → фрагментация доски растёт быстрее, чем её гасят очистки → закономерный тупик. Одновременно таймер делает важными скорость и точность (новый слой мастерства).
- Бот: на каждом уровне укладывается в текущий `turnTime`; на низких уровнях имитирует «человеческие» паузы.

> Вывод по риску из п. 13: при включённом таймере «до тупика» — корректное и конечное условие. «Лимит ходов/времени» остаётся как более мягкая опция для совсем казуальных/детских столов.

### 2.8. Режимы игры
| Режим | Описание | Фаза |
|---|---|---|
| **vs Bot** | Игрок против ИИ (3 уровня) | MVP |
| **Hot-seat (локальный)** | Два игрока по очереди на одном устройстве | Фаза 2 |
| **Online PvP** | Матчмейкинг, общая доска, ходы через сервер | Фаза 3 |
| **Practice / Solo** | Тренировка без соперника, на рекорд | опц. |

### 2.9. Детерминизм (требование ко всему ядру)
Состояние партии полностью определяется: `matchSeed` + конфигом правил + журналом ходов. Один и тот же вход → один и тот же исход. Нужно для: воспроизведения реплеев, синхронизации онлайн, автотестов и анти-чита.

## 3. Искусственный интеллект (бот)

Бот видит **только публичную информацию**: текущую доску, свою руку, общий превью следующих фигур. Бот **не читает зерно** мешка (не «жульничает»).

### 3.1. Общая схема
Для каждого уровня бот перебирает все допустимые `(фигура из руки, позиция, поворот)` и выбирает ход по функции оценки `evaluate(boardAfterMove)`.

Признаки оценки (feature vector):
- `clears` — число очищенных единиц за ход (вес **+**, сильный);
- `holes` — пустые клетки, окружённые заполненными/трудные для заполнения (вес **−**);
- `nearLines` — строки/столбцы/боксы, которым не хватает 1–2 клеток (вес **+**, заготовки);
- `fragmentation` — разрозненность заполнения (вес **−**);
- `boxProgress` — заполненность боксов (вес **+**);
- `oppThreat` (только Hard) — насколько ход дарит сопернику лёгкую очистку (вес **−**).

```
score = w1*clears + w2*nearLines + w3*boxProgress
      - w4*holes - w5*fragmentation - w6*oppThreat
```

### 3.2. Уровни сложности
| Уровень | Поведение | Алгоритм |
|---|---|---|
| **Тупой (Easy)** | Случайный валидный ход; иногда берёт очевидную очистку с вероятностью p≈0.3 | Random valid + слабый greedy |
| **Умный (Medium)** | Жадная эвристика на 1 ход вперёд по `evaluate` | 1-ply greedy, веса сбалансированы |
| **Сложный (Hard)** | Поиск на 1–2 хода + учёт ответа соперника + денай его очисток | Expectimax/мини-макс по руке + сэмплинг будущего |

### 3.3. Псевдокод (Medium)
```
function chooseMove(board, hand):
    best = null
    for piece in hand:
        for rot in rotations(piece):
            for pos in validPositions(board, piece, rot):
                b2 = simulate(board, piece, rot, pos)   # с очисткой
                s  = evaluate(b2)
                if best == null or s > best.score:
                    best = {piece, rot, pos, score: s}
    return best
```

### 3.4. Псевдокод (Hard, упрощённо)
```
function chooseMoveHard(board, hand, previewPieces):
    bestScore = -inf; bestMove = null
    for move in allMoves(board, hand):
        b2 = simulate(board, move)
        myGain = scoreGained(move)
        # ответ соперника: его лучший ход на b2
        oppBest = max over move2 in allMoves(b2, samplePieces(previewPieces)):
                      scoreGained(move2) + 0.5*evaluate(simulate(b2, move2))
        net = myGain + 0.7*evaluate(b2) - 0.6*oppBest
        if net > bestScore: bestScore = net; bestMove = move
    return bestMove
```

### 3.5. Тайминг и «человечность»
- Бот делает ход с задержкой 0.4–1.2 с (анимация «думает»).
- Easy иногда «ошибается» намеренно; Hard может имитировать паузы на сложных позициях.

- Прогрессивные ачивки (например, «очисти 1 000 линий суммарно»).
- Хранятся локально (MVP) → синхронизация на сервере (Фаза 3).

## 6. Архитектура и технологии

### 6.1. Ключевой принцип — отделяемое логическое ядро
Вся игровая логика (доска, мешок, ходы, очистка, очки, конец партии, бот) живёт в **чистом, платформо-независимом модуле** без зависимостей от UI и сети. Ядро детерминированно и покрыто тестами. Презентация и сеть — отдельные слои. Это позволяет:
- переиспользовать ядро на клиенте и сервере (валидация ходов на сервере = тот же код);
- легко тестировать симуляциями;
- менять движок рендеринга без переписывания правил.

```
┌─────────────────────────────────────────────┐
│                 Презентация                   │  ← рендер, ввод, анимации
├─────────────────────────────────────────────┤
│              Игровое ядро (pure)              │  ← правила, бот, детерминизм
├─────────────────────────────────────────────┤
│   Платформенные сервисы / Сетевой слой        │  ← хранилище, аккаунт, WebSocket
└─────────────────────────────────────────────┘
```

### 6.2. Рекомендация по стеку (для MVP «ядро + бот», кроссплатформа)
| Вариант | Плюсы | Минусы | Когда выбрать |
|---|---|---|---|
| **Flutter + Flame** (рекоменд.) | Один код → web/desktop/mobile, хорошая анимация, быстрый старт | Менее «игровой», чем движок | Приоритет — скорость и единая база |
| **Godot 4** | Полноценный 2D-движок, экспорт во все цели, бесплатный | Web-экспорт чуть тяжелее | Нужен максимум «сочности»/эффектов |
| **TypeScript + React (web) / RN (mobile) / Tauri (desktop)** | Веб-экспертиза, гибкость | Фрагментация платформ | Если команда — веб-разработчики |

> Логическое ядро в любом случае пишется как изолированный модуль (Dart / GDScript / TypeScript). Рекомендуемый путь MVP: **Flutter + Flame**, web-сборка как первая витрина.

### 6.3. Слой сети (Фаза 3, онлайн)
- Модель: **server-authoritative**, ход валидируется тем же ядром на сервере. Реального времени/физики нет — пошаговость делает синхронизацию дешёвой.
- Транспорт: **WebSocket** (события: `match_found`, `move`, `state_update`, `game_over`).
- Сервер: **Node.js (NestJS)** или **Go**; **Redis** для матчей/сессий; **PostgreSQL** для профилей, ачивок, лидербордов.
- Матчмейкинг: очередь по MMR; приватные комнаты по коду.

### 6.4. Локальное хранилище
- MVP: встроенная БД/хранилище (SQLite / local storage) для профиля, ачивок, настроек, сохранённой партии.
- Фаза 3: фоновая синхронизация при наличии аккаунта.

- **Игровой экран:** доска 9×9, рука каждого игрока, счёт обоих, индикатор очереди хода, превью следующих фигур, кнопки (поворот, пауза).
- **Экран результата:** победитель, счёт, полученный XP/ачивки, «Реванш» / «В меню».
- **Профиль / Ачивки / Настройки / Магазин.**

### 8.2. Игровое взаимодействие
- Drag-and-drop фигуры на доску с «призраком» (предпросмотр позиции) и подсветкой будущих очисток.
- Невалидная позиция подсвечивается; нельзя зафиксировать.
- Тап по фигуре в руке для поворота (если включён).
- «Juice»: анимация и звук очистки, частицы, всплывающие очки, экранная встряска на больших комбо.
- Чёткая индикация «чей ход».

### 8.3. Доступность
- Фигуры различимы по форме и паттерну (не только по цвету).
- Крупные тач-зоны, поддержка отмены неудачного перетаскивания (drop вне доски = возврат).

## 11. Безопасность и анти-чит (для онлайн, Фаза 3)

## 14. Приложение A. Ключевые алгоритмы (псевдокод)

### A.1. Проверка и применение очистки
```
function resolveClears(board):
    rowsFull = [r for r in 0..8 if all(board[r][c].filled for c in 0..8)]
    colsFull = [c for c in 0..8 if all(board[r][c].filled for r in 0..8)]
    boxesFull = [b for b in 0..8 if boxFilled(board, b)]
    N = len(rowsFull) + len(colsFull) + len(boxesFull)
    cellsToClear = unionOfCells(rowsFull, colsFull, boxesFull)
    for cell in cellsToClear: board[cell].filled = false
    return N        # число очищенных единиц
```

### A.2. Очки за ход
```
function scoreForMove(N, comboCounter, isPerfectClear):
    if N == 0: return 0
    base = N * (N + 1) / 2
    multiplier = 1 + 0.1 * min(comboCounter, 10)
    bonus = isPerfectClear ? 15 : 0
    return round(base * multiplier) + bonus
```

### A.3. Проверка тупика (конец партии)
```
function hasAnyMove(board, hand):
    for piece in hand:
        for rot in rotations(piece):
            for pos in allCells:
                if canPlace(board, piece, rot, pos): return true
    return false
# Если hasAnyMove(board, currentPlayer.hand) == false → партия завершена.
```

## 17. Видение v2 — расширение скоупа

Стратегия продукта на горизонте 12–18 месяцев после релиза MVP. Подходим к
BlockDuel не как к одной игре, а как к **PvP-платформе для классических
головоломок с квадратными фигурами**: 9×9, тетрис, мемори, 3-в-ряд и т.д.
Общее ядро (сетка + ходы по очереди + scoring) переиспользуется; режимы
подключаются как самостоятельные модули.

### 17.1. Глубокая прогрессия и квестовая система

**Уровни 1–100.** Каждому уровню соответствует именованный титул и одна
гарантированная награда:

| Диапазон | Награда |
|---|---|
| 1–10 (Новичок) | Базовые скины ячеек, дополнительный power-up |
| 11–25 (Знаток) | Альтернативная тема, +1 ежедневный quest |
| 26–50 (Мастер) | Эксклюзивный аватар-кадр, ELO-сезонный значок |
| 51–75 (Эксперт) | Доступ к ranked-режимам, особый эффект очистки |
| 76–99 (Чемпион) | Прозрачные темы UI, prestige-аватарка |
| **100** | **Разблокировка специального набора фигур** (см. п. 17.10) |

**Квесты.** Три параллельных уровня:

- **Daily (3–5 шт.):** легкие условия, ротация в 00:00 UTC, награда 50–150
  кристаллов и 200–400 XP.
- **Weekly (1–2 шт.):** среднее усилие, тематика недели, награда
  500–1500 кристаллов, эпик-скин.
- **Seasonal/Event (1 шт.):** требует от 5 до 20 матчей, награда —
  prestige-предмет (эксклюзивная тема, аватар, музыка).

Квесты не дублируют ачивки: ачивки — постоянные «трофеи», квесты —
повторяющийся eng cyclic loop.

### 17.2. Режим **Puzzle** — сборка силуэтов

Соло-режим без таймера и противника. Игрок собирает заранее
определённый силуэт (кот, медведь, пони, дельфин, …) из ограниченного
набора фигур. Цель — закрасить все клетки шаблона ровно за N ходов.

Технически:
- Шаблон — двумерная маска на доске 9×9 (или 12×12 для крупных силуэтов).
- Hand содержит ровно те фигуры, что нужны для решения (без избытка).
- Зона вне маски заблокирована и подсвечена как «не использовать».
- Каждый уровень собирается в один правильный путь, но фигуры можно
  ставить в любом порядке.

**Не-стандартные piece-set** (отдельно от 7 тетромино классической игры):

- L-shape с **закруглённым** наружным углом.
- T-shape со **скошенным** хвостом.
- Диагональные «зубчатые» фигуры (для коврового рисунка).
- Mini-piece (1 клетка), Plus-piece (+ 5 клеток), Pentomino-варианты.

Эти piece-set'ы изолированы от main-game (где играют классические
тетромино) и не попадают в 7-bag в PvP-режимах.

### 17.3. Локальный мультиплеер — **LAN PvP**

Игра двух игроков на двух устройствах в одной локальной сети **без
интернет-соединения** (самолёт, поезд, дача).

Каналы (по убыванию приоритета):

| Канал | Протокол | Когда выбирать |
|---|---|---|
| **Wi-Fi Direct / hotspot** | mDNS + WebSocket поверх локального LAN | оба устройства в одной Wi-Fi сети |
| **BLE peer-to-peer** | Bluetooth LE + JSON-frame | нет общего Wi-Fi (самолёт) |
| **QR pairing fallback** | один создаёт hotspot, второй подключается | худший сценарий |

Авто-обнаружение через advertising/discovery. Один игрок создаёт комнату
(host), второй сканирует и подключается. Сам матч — те же server-authoritative
правила, но «сервер» физически запущен на одном из устройств. Лидерборд
LAN-матчей **не** учитывается в глобальном ELO.

### 17.4. PvP-расширения — новые режимы поверх ядра

Главный концепт продукта: **превратить классические одно-игровые
головоломки в pvp**. Минимальный набор v2:

#### 17.4.1. Memory Solo
Игрок видит финальную раскладку фигур на доске 3 секунды → доска
очищается → игрок должен воспроизвести расположение. Скоринг по
точности + время.

#### 17.4.2. Memory Duel
Игрок A расставляет N фигур на своей доске → доска показывается игроку B
3–5 секунд → доска очищается → игрок B воспроизводит. Затем роли
меняются. Победитель — кто точнее.

#### 17.4.3. Co-op Tetris (turn-based на одном поле)
Одно поле, два игрока по очереди ставят тетромино. Очки за очистку идут
тому, **чей ход вызвал очистку** (как в 9×9). Поле классическое тетрисное
(10×20), без падающих фигур — pure puzzle.

#### 17.4.4. Match-3 PvP (зарезервировано)
Поле 8×8 c цветными элементами, ход = swap двух соседних. Очки за серии
≥ 3 одного цвета. Хорошо ложится на нашу turn-based архитектуру.

Каждый режим имеет:
- Отдельный реализуемый модуль `src/modes/<mode-id>/`.
- Свои правила (ядро game-logic), reuse общих UI-компонентов.
- Свой рейтинг и набор ачивок.

### 17.5. Социальная система

**Друзья.**
- Поиск по нику, ID, или QR-коду.
- Список друзей с онлайн-статусом, последним рейтингом, общим прогрессом.
- Прямое приглашение в матч с заданным режимом и cfg.

**Управление матчем.**
- Кнопка «**Предложить ничью**» доступна с 10-го хода. Соперник
  принимает/отклоняет; при принятии — XP и ELO начисляются как ничья.
- Кнопка «**Сдаюсь**» — мгновенное завершение, поражение для сдавшегося.
- Кнопка «**Завершить досрочно**» с auto-judgement по текущему счёту.

**Антиабуз.**
- **Quit-penalty:** выход во время своего хода без resign/draw-offer →
  −10 ELO + штраф в виде 24h cooldown на ranked. Бот-fallback на сервере
  доигрывает матч, не давая сопернику терять время.
- **AFK-penalty:** 3 раза подряд force-place из-за таймаута → авто-resign.
- **Rage-quit detection:** 3 раза quit-penalty за 24 часа → 7-дневный
  ranked-suspension.

### 17.6. XP, рейтинги, общий зачёт

**XP-формула за матч:**
```
xp = floor(baseXp · resultMult · diffMult · streakBonus)
```
- `baseXp` — базовая ставка матча (60 для 9×9, 80 для Co-op Tetris и т.д.).
- `resultMult` — победа `1.0`, ничья `0.5`, поражение `0.0`.
- `diffMult` — `0.7` (Тупой бот) … `1.0` (Hot-seat / равный online) …
  `1.4` (Hard бот) … `1.6` (ranked online против ≥ +50 ELO).
- `streakBonus` — `1 + 0.05 · win_streak` (cap 1.5).

**Composite score.** Общий счёт игрока = взвешенная сумма ELO во всех
ranked-режимах + XP-уровень × множитель ладдера. Отображается в профиле
крупно как «титул» (Bronze/Silver/Gold/Platinum/Diamond/Master).

**Ladder-листы.**

| Ладдер | Источник |
|---|---|
| **General** | Composite score |
| **9×9 PvP** | ELO в основном режиме |
| **Bot Mastery** | Win-rate против Hard-бота за сезон |
| **Arcade** | High-score в соло-арена |
| **Memory** | ELO в Memory Duel |
| **Co-op Tetris** | ELO |
| **Puzzle Speedrun** | Лучшее время прохождения season-набора |

Сезон = 90 дней; в конце top-100 получают prestige-награды; ELO у
остальных мягко сжимается к 1000.

### 17.7. Быстрая игра (Quick Play)

Главная кнопка стартового экрана. Один тап → попадание в матч с
дефолтными настройками (handSize=3, rotation+flip on, blitz norm,
тема — текущая) против random-online-оппонента или Medium-бота при
fallback'е. Quick PvP пропускает SetupScreen для онбординг'а скорости.

### 17.8. Магазин v2 и кастомизация

**Валюта.**
- **Кристаллы** — премиум, покупаются за реальные деньги
  (pack-of-100/500/1500/5000 + бонус за объём) или зарабатываются медленно
  (по 5–15 за квест).
- **Coins** — мягкая валюта, начисляется за каждый матч и очистку.

**Категории товаров:**

| Категория | Кристаллы | Coins |
|---|---|---|
| Скины ячеек (block-styles) | да | да |
| Темы интерфейса (full pack) | да | — |
| Фоны (мир, природа, неон) | да | да |
| Эффекты очистки (взрыв, искры, конфетти) | да | — |
| Музыкальные паки | да | — |
| Аватары и кадры | да | да |
| Power-ups (расходники) | — | да |
| Сезонный pass | да | — |

**Видимость скинов соперника.** По умолчанию **противник видит**, что мы
играем — он смотрит на свою копию доски, на которую транслируется наш
скин ячеек / эффект очистки (server рассылает skinId в state). Это даёт
покупкам real social-value.

Исключение: режим **«адаптивный/доступный»** (accessibility) — у этого
игрока всё рендерится в максимально контрастной нейтральной палитре,
скины соперника не применяются.

**Создание своей темы.** Премиум-инструмент (5000 кристаллов): выбор
двух цветов игроков (P0/P1), фона, шрифта; превью и сохранение.
Опционально публикация в маркетплейс для одобрения модераторами.

### 17.9. Расширяемая семья досок

Доска — не константа `9×9`, а параметризованная сущность:

| Mode | Field | Pieces |
|---|---|---|
| **BlockDuel 9×9** (default) | 9×9 + 3×3 boxes | 7 классических тетромино |
| **Classic Tetris PvP** | 10×20 | те же тетромино, без падения |
| **Match-3 8×8** | 8×8 цвета | 6 цветных «леденцов» |
| **Puzzle Silhouette** | mask N×M | специальный piece-set с закруглениями |
| **Memory** | 9×9 | те же тетромино, но визуализация |
| **(slot)** | custom | для будущих режимов |

Архитектурно: общий interface `GameRules` + per-mode реализация.

### 17.10. Гарантия валидного старта (низкие уровни)

Для игроков **≤ 10 уровня** при формировании стартовой руки гарантируем,
что **хотя бы одна** фигура из руки имеет валидный ход на пустой доске.
Это исключает фрустрирующий сценарий «открыл партию — рука сразу dead».

Реализация: после `Bag.drawAvoiding(handSize)` проверяем
`hasAnyMove(board, hand, cfg)`; если false — реролл bag-snapshot до 3
попыток с тем же seed для записи в реплей, иначе fallback на любой valid
piece из 7-bag.

Для уровней > 10 защита отключается — высокий уровень предполагает
ситуации с «плохой рукой».

### 17.11. Единый аккаунт и кросс-устройство

Уже частично есть в Фазе 4 (Google sign-in + Firestore sync). К v2:

- Привязка email + Google + Apple ID + Telegram OAuth → один игрок может
  входить с любого из устройств без потери прогресса.
- Active-session-management: при логине на новом устройстве — уведомление
  на старое, опция forced-logout.
- Конфликт-резолвер для офлайн-сессий: при последующем входе мерж
  XP/Coins/Achievements по схеме «max wins».

### 17.12. Пасхалки и секретные мини-игры

- **Hidden modes** в главном меню по комбо-тапу логотипа: snake поверх
  9×9, classic block-on-block.
- **Сезонные ивенты** (Halloween, Новый год): тематические скины,
  специальные доски-снежинки.
- **Achievement-triggered Easter eggs:** после 100 wins подряд — секретная
  мини-игра «Boss Fight» против специального бота.
- **Console codes** (для разработчика и dev-сообщества) активируют debug
  HUD, FPS-meter, дамп state в clipboard.
