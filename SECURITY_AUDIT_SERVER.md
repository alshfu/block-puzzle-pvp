# Аудит безопасности PvP-сервера BlockDuel 9×9

**Дата:** 2026-06-09 · **Объём:** `server/{index,lobby,room,leaderboard,types}.ts`
(Node.js + `ws`, server-authoritative). **Метод:** статический разбор кода +
threat-model протокола WS; активные тесты — отдельный план (см. §5), против
прод-VPS без согласования и окна обслуживания **не запускались**.

> ⚠️ Прод `pvp.alshfu.com` делит VPS со сторонним сайтом пользователя
> (`alshfu-arena.conf`). Любое активное нагрузочное/DoS-тестирование может задеть
> соседний сервис — выполнять только в окне и только по явному согласованию,
> предпочтительно против локального/стейджингового инстанса.

## 1. Резюме

Игровая механика защищена хорошо: сервер авторитетен, ходы валидируются
(очередь, фигура в руке, `canPlace`, **анти-чит ориентаций** через
`orientations()`), счёт считается на сервере. Подделать игровой результат
«снизу» (через клиента-читера) трудно.

Основные риски — **не в геймплее, а в инфраструктуре протокола**: отсутствует
аутентификация личности (id игрока самозаявляемый), нет ограничений размера
сообщений и частоты, нет лимитов на число комнат/очередь/записи лидерборда,
синхронная запись файла на горячем пути. Это открывает **спуфинг/угон слота,
отравление ELO-лидерборда и несколько векторов DoS**.

| Severity | Кол-во |
|----------|--------|
| High     | 5 |
| Medium   | 6 |
| Low      | 4 |

Бо́льшая часть находок уже исправлена тремя раундами **SEC-1 / SEC-2 / SEC-3**
(2026-06-09, см. §7). Статус у каждой — пометкой в заголовке (**[ИСПРАВЛЕНО
SEC-N]** / **[ЧАСТИЧНО]**). Закрыты все High (H1/H3/H4/H5 полностью, H2 частично)
и большинство Medium/Low. Открытыми по существу остаются M6 (спуфинг ника —
санитизация при выводе), L3/L4 и полный анти-абьюз ELO — последнее требует
внешней идентичности (Firebase-uid) и вынесено в «осталось» §7.

## 2. Сильные стороны (подтверждено)

- **Server-authoritative gameplay.** `handleMove` проверяет `status==='playing'`,
  `idx===current`, наличие `pieceId` в руке, `canPlace`, и **валидную ориентацию**
  (`normalize(cells)` ∈ `orientations(type, rotation, flip)`). Счёт/перебор —
  на сервере (`scoreMoveDetailed`, `findClears`, `hasAnyMove`). Клиент не может
  навязать очки/нелегальную фигуру.
- **Изоляция комнат.** Подключение к `room` отвергается, если комната не создана
  через лобби (`"room not initialised"`).
- **Грациозная обработка парс-ошибок.** `JSON.parse` в `try/catch` везде —
  битый JSON не роняет процесс.
- **Корректная очистка ресурсов при штатном закрытии.** Таймеры комнаты/лобби
  останавливаются, пустая комната удаляется (`onAllGone`).

## 3. Находки

### HIGH

**H5 — [ИСПРАВЛЕНО SEC-1] Краш всего процесса от одной ошибки сокета.**
*Где:* `index.ts` `wss.on("connection")` — у соединений не было обработчика
`error`. *Суть:* найдено активным локальным тестом: один кадр сверх `maxPayload`
(или любая ошибка сокета) эмитит unhandled `'error'` → Node роняет **весь
процесс** (DoS одним сообщением). *Импакт:* критический — мгновенная
недоступность (и соседнего сайта на VPS). *Фикс (внедрён):* на каждое соединение
вешается `ws.on("error", () => ws.close())` — гасим на уровне сокета. ⚠️ Это был
скрытый баг и до `maxPayload`; теперь закрыт.

**H1 — [ИСПРАВЛЕНО SEC-2, под флагом] Отсутствие аутентификации личности (спуфинг id).**
*Где:* `room.handleHello` (`players.findIndex(p => p.profile.id === profile.id)`),
весь протокол. *Суть:* `OnlineProfile.id` — самозаявляемый UUID от клиента, без
токена/подписи. Любой клиент может прислать чужой `id`. *Импакт:* подделка
личности; основа для H2/H3. *Фикс:* ввести аутентификацию (подписанный токен
сессии от лобби при `matched`, проверяемый в `hello`; либо привязка к Firebase
ID-токену, который уже есть на клиенте). Минимально — выдавать в `matched`
одноразовый `roomToken`, привязанный к слоту, и требовать его в `hello`.

**H2 — [ЧАСТИЧНО SEC-2/3] Отравление ELO-лидерборда / фарм рейтинга.**
*Статус:* roomToken (H1) закрывает воздействие на **чужой** id (нельзя подставить
жертву). Само-фарм своими id всё ещё возможен — нужен анти-абьюз (rate-limit
матчей, запрет «сам против себя», мин. длина матча).
*Где:* `leaderboard.reportMatch`, `lobby` matchmaking. *Суть:* т.к. личность не
аутентифицирована, можно (а) фармить ELO, сводя два своих подключения в матч и
сдаваясь нужной стороной; (б) влиять на рейтинг **чужого** id, подставив его как
участника. Записи создаются для любого присланного id (`getOrCreate`). *Импакт:*
полная компрометация честности лидерборда. *Фикс:* аутентификация (H1) + анти-
абьюз: не засчитывать матч между одинаковым «человеком»/IP, минимальная длина
матча, rate-limit на матчи/час на id, идемпотентность отчётов.

**H3 — [ИСПРАВЛЕНО SEC-1] Угон слота активного матча (session hijack).**
*Где:* `handleHello` безусловно `players[idx].conn = conn`. *Суть:* зная `roomId`
и `id` участника, третье соединение шлёт `hello` и **перехватывает слот** —
ссылка `conn` заменяется, прежний игрок «обрезается». *Импакт:* гриф (выкинуть
игрока), перехват матча. *Фикс:* отклонять `hello`, если слот уже занят живым
`conn` (или требовать roomToken из H1); опционально — уведомлять/закрывать
старый conn только если он мёртв (`readyState!==OPEN`).

**H4 — [ИСПРАВЛЕНО SEC-1] DoS: нет лимита размера сообщений и частоты (flood).**
*Где:* все `conn.on("message")` + `WebSocketServer({ noServer:true })` без
`maxPayload`. *Суть:* нет ограничения размера кадра (по умолчанию `ws` — до
100 MiB) и нет rate-limit. Огромный `move.cells` проходит через `JSON.parse` +
`normalize` + `JSON.stringify` → CPU/RAM. Поток сообщений не ограничен. *Импакт:*
исчерпание CPU/памяти, недоступность (и для соседнего сайта на VPS). *Фикс:*
`new WebSocketServer({ noServer:true, maxPayload: 16*1024 })`; per-conn rate-limit
(токен-бакет, напр. 30 msg/сек); ранняя валидация длины `cells` (≤ макс. размера
фигуры, 4).

### MEDIUM

**M1 — [ИСПРАВЛЕНО SEC-1] Нет валидации формы/диапазона `move` (cells/r/c).**
*Где:* `handleMove`. *Суть:* `cells: Coord[]`, `r`, `c` приходят сырыми; длина
`cells` и тип/диапазон координат не проверяются до `normalize`/`canPlace`. Хотя
`canPlace`/анти-чит отвергнут невалидное, обработка дорогих/странных входов идёт
до отказа; нецелые/огромные значения — риск исключения. *Фикс:* схема-валидация
входа (целые в [0,8], `cells.length` 1..4, `r/c` целые в диапазоне) до логики.

**M2 — [ЧАСТИЧНО SEC-1] Неограниченный рост числа комнат и очереди лобби.**
*Статус:* добавлен потолок очереди `MAX_QUEUE=500`; лимит числа одновременных
комнат и TTL пустых `waiting`-комнат — ещё нет.
*Где:* `index.ts` `rooms`, `lobby.queue`. *Суть:* нет потолка на число
одновременных комнат/длину очереди → массовое создание матчей/постановок в
очередь исчерпывает память. *Фикс:* лимиты (макс. комнат, макс. очередь),
backpressure, TTL пустых `waiting`-комнат.

**M3 — [ИСПРАВЛЕНО SEC-3] Неограниченный рост лидерборда (память + файл).**
*Где:* `leaderboard.entries`, `save()`. *Суть:* запись создаётся для любого
нового id (`getOrCreate`); флуд уникальными id раздувает Map и JSON-файл. *Фикс:*
аутентификация (H1) ограничивает создание; плюс предел числа записей/эвикшен
неактивных, отдельная БД вместо JSON при росте.

**M4 — [ИСПРАВЛЕНО SEC-3] Синхронная запись файла на горячем пути + неатомарно.**
*Где:* `leaderboard.save()` (`writeFileSync` на каждый матч). *Суть:* блокирует
event-loop под нагрузкой (усиливает DoS); неатомарная запись → при крахе
возможен повреждённый JSON, а `load()` при ошибке парсинга **теряет все записи**.
*Фикс:* асинхронная запись с дебаунсом; атомарно (write temp + `rename`);
бэкап/версионирование файла.

**M5 — [ИСПРАВЛЕНО SEC-3, под env] Нет проверки Origin при WS-upgrade (CSWSH).**
*Где:* `httpServer.on("upgrade")`. *Суть:* WebSocket не подчиняется SOP/CORS;
любая веб-страница может открыть соединение к серверу. Без cookie-аутентификации
прямой кражи сессии нет, но это даёт любому сайту драйвить протокол (усиливает
H3/H4). *Фикс:* проверять заголовок `Origin` против allow-list доменов на
upgrade; иначе `socket.destroy()`.

**M6 — Спуфинг ника/аватара в лидерборде (потенц. stored XSS).**
*Где:* `reportMatch` перезаписывает `nick`/`avatar` из отчёта без санитайза/
лимита длины. *Суть:* произвольный/огромный ник сохраняется и рассылается всем
подписчикам. Flutter рендерит как `Text` (низкий риск), но любой будущий
web/HTML-рендер лидерборда → stored XSS; длинный ник — раздувание. *Фикс:*
валидация длины (≤24) и набора символов ника на сервере; экранирование на любых
HTML-поверхностях.

### LOW

**L1 — [ИСПРАВЛЕНО SEC-3] Предсказуемый `roomId` (`Math.random` + время).**
*Где:* `index.ts`. *Суть:* `m_<base36 time>_<6 симв.>` слабо случайный; помогает
энумерации комнат (вкупе с H1/H3). *Фикс:* `crypto.randomUUID()`.

**L2 — [ИСПРАВЛЕНО SEC-3] Не-крипто `Math.random()` для `matchSeed`.**
*Где:* `index.ts`, `room.handleRematch`. *Суть:* раздача предсказуема при знании
seed; для соревновательной честности желательна непредсказуемость. *Фикс:*
`crypto.randomInt`. (Низкий — детерминизм нужен ядру, но seed может быть крипто.)

**L3 — Не-участник держит соединение к комнате.**
*Где:* `room.handleConnection` принимает любой conn; `hello` чужака отвергается,
но соединение остаётся открытым и его сообщения парсятся (no-op). *Суть:* мелкий
ресурсный/флуд-вектор. *Фикс:* закрывать conn при первом не-`hello`/неуспешном
`hello`; таймаут на `hello`.

**L4 — `/healthz` без ограничений и информации о версии.**
*Где:* `index.ts`. *Суть:* открытый эндпойнт (ок), но стоит убедиться, что не
отдаёт лишнего и не используется как амплификатор. *Фикс:* оставить «ok», без
деталей; rate-limit на уровне nginx.

## 4. Приоритеты исправления

1. **H4 + M1** (быстро, максимум эффекта): `maxPayload`, rate-limit, валидация
   входа `move`. Закрывает основной DoS малой кровью.
2. **H1 → H2/H3**: roomToken/аутентификация. Закрывает спуфинг, угон слота и
   отравление лидерборда (главные угрозы целостности).
3. **M4 + M3**: асинхронный атомарный save + лимиты/эвикшен лидерборда.
4. **M2 + M5**: лимиты комнат/очереди, проверка Origin.
5. **L1/L2**: `crypto` для id/seed. **M6/L3/L4**: санитайз ника, гигиена conn.

## 5. План активного пентеста (запускать локально/стейджинг)

Против локального сервера (`npm run server:dev`, `PORT=1999`), не против прода:

- **Fuzzing протокола:** битый/частичный JSON, неизвестные `type`, отсутствующие
  поля, неверные типы — сервер не должен падать (ожидаемо ок).
- **Oversized payload:** кадр 1–50 MiB на lobby/room — проверить отказ/лимит
  (сейчас лимита нет → H4).
- **Message flood:** N msg/сек на одно соединение — наблюдать CPU/латентность.
- **Identity spoof / slot hijack:** создать матч, затем третьим соединением
  прислать `hello` с id участника — убедиться, что слот **не** угоняется
  (сейчас угоняется → H3).
- **Leaderboard poisoning:** свести два своих conn в матч, сдаться — проверить
  изменение ELO; подставить чужой id участником.
- **Unique-id flood:** очередь/матчи с тысячами уникальных id — рост памяти/файла
  (M2/M3).
- **Origin check:** upgrade с произвольным `Origin` — должен отклоняться (сейчас
  нет → M5).

Безопасный нес-деструктивный харнесс: `tools/pentest_local.mjs` (см. ниже) —
прогоняет подмножество с малыми лимитами, печатает наблюдаемое поведение. Перед
прогоном поднять локальный сервер.

```bash
npm run server:dev                 # терминал 1 (PORT=1999)
node tools/pentest_local.mjs       # терминал 2
```

## 7. Внедрённые фиксы — SEC-1 (2026-06-09)

Серверные, без изменения протокола → **безопасны для прод-деплоя** (живой TS-
клиент на Pages продолжит работать). Подтверждено локальным прогоном
`tools/pentest_local.mjs`.

- **H5** — `ws.on("error")` на каждое соединение (`index.ts`): oversized/любая
  ошибка больше **не роняет процесс** (подтверждено: T2 закрывает сокет 1009,
  сервер жив).
- **H4** — `maxPayload: 16 KiB` на `WebSocketServer` + `RateLimiter`
  (40 msg/1с на conn) в лобби/комнате/лидерборде (`server/limits.ts`).
- **H3** — `handleHello` отклоняет занятый ЖИВОЙ слот (`"slot already
  connected"`); реконнект на мёртвый conn по-прежнему разрешён (подтверждено:
  T5 «перехват слота отклонён»).
- **M1** — `isValidMoveInput` (cells 1..5 целых пар в [0,8], r/c в [0,8]) до
  игровой логики; `pieceId` ограничен длиной.
- **M2** (частично) — `MAX_QUEUE=500` в лобби.
- **M6** (частично) — `isValidProfile` (длины id≤64/nick≤24/avatar≤8) на входе
  лобби; `myId` лидерборда ограничен 64.

**Деплой SEC-1:** на VPS `git pull && systemctl restart blockduel-pvp` (делает
пользователь). Откат тривиален (только server/*).

### SEC-2 (2026-06-09) — аутентификация roomToken (end-to-end)

- **Протокол:** `matched` несёт одноразовый `token` слота (server → client),
  `hello` принимает `token?` (client → server). `party/protocol.ts`.
- **Сервер:** лобби генерит `randomUUID()` на слот (`server/lobby.ts`), Room
  хранит `[tokenA, tokenB]` и валидирует в `handleHello` (`server/room.ts`):
  прислан токен → обязан совпасть; не прислан → отказ только при
  `REQUIRE_ROOM_TOKEN=1`. Закрывает H1, угон чужого слота и подстановку чужого
  id в матч/лидерборд (H2 частично).
- **Клиенты (оба):** Flutter (`lobby_notifier`→`OnlineGameArgs`→`OnlineMatchArgs`
  →`hello`) и **TS** (`client.ts`→`OnlineMenuScreen`→`App`→`OnlineGameScreen`→
  `useOnlineGame`→`hello`) пробрасывают token из `matched` в `hello`.
- **Раскатка без даунтайма (флаг `REQUIRE_ROOM_TOKEN` default off):**
  1. Задеплоить новые клиенты (Pages + Flutter) — они шлют token, старый сервер
     его игнорирует. Безопасно.
  2. Задеплоить сервер — токены выдаются и валидируются (флаг off → старые
     клиенты ещё работают без token).
  3. Когда все клиенты обновлены — выставить `REQUIRE_ROOM_TOKEN=1` в systemd
     unit и `systemctl restart blockduel-pvp` → жёсткое требование.
- **Подтверждено** `tools/pentest_local.mjs` (T9): флаг off — верный→joined,
  неверный→отказ, без токена→joined; флаг on — без токена→«auth required».
  Краша процесса нет.

### SEC-3 (2026-06-09) — добивка серверных хвостов

Серверные, без смены протокола → безопасно для прод-деплоя.

- **M3/M4** (`server/leaderboard.ts`): запись лидерборда теперь **async +
  атомарная** (debounce 1с → temp-файл → `rename`), не блокирует event-loop и
  не бьёт JSON при крахе; **эвикшен** старейшей записи при `MAX_ENTRIES=50000`
  (анти-флуд уникальными id).
- **M5** (`server/index.ts`): проверка `Origin` на upgrade против
  `ALLOWED_ORIGINS` (CSV в env). Пусто → разрешены все (прод не ломается);
  запросы без Origin (нативные клиенты) всегда проходят. Подтверждено харнессом
  (off→принят, on→403).
- **M2** (`server/index.ts`): потолок `MAX_ROOMS=2000` + TTL `120с` для пустых
  `waiting`-комнат (подчистка, если клиенты не пришли).
- **L1/L2**: `roomId` = `crypto.randomUUID()`, `matchSeed`/seed реванша =
  `crypto.randomInt` (непредсказуемо; вместо `Math.random`/времени).
- **H2** (частично, `server/leaderboard.ts`): матч «сам против себя» (одинаковый
  id у обоих участников) не меняет ELO/W-L-D. Антифарм РАЗНЫМИ аккаунтами
  по-прежнему требует реальной идентификации (вне in-memory сервера).

**Осталось (опц., требует внешней идентичности/инфры):** полный анти-абьюз ELO
(rate-limit матчей/час на личность, привязка к Firebase-uid); вынос лидерборда в
БД при значительном росте.

## 8. Что НЕ делалось и требует решения пользователя

- Активные тесты против **прод** `pvp.alshfu.com` (нагрузка/DoS/спуфинг) — не
  запускались (риск для соседнего сайта на VPS). Нужно явное согласие + окно
  обслуживания, лучше — стейджинг-копия.
- Внедрение фиксов в `server/*.ts` — по согласованию (потребуется
  `git pull && systemctl restart blockduel-pvp` на VPS, это делает пользователь).
